## 1. Purpose

This document defines the first landing-page direction for `punk`.

The landing should not pretend the product is already complete. Its job is to:

- explain the concept clearly
- present the public build story
- attract early believers, collaborators, angels, and future users
- introduce the mascot and visual language
- avoid overclaiming
- create a stable design direction before implementation starts

The first landing is a public-build artifact, not a full product website.

## 3. Design direction

## Chosen direction: Premium Cypherpunk Kernel

The landing should combine:

- cypherpunk privacy/protocol energy
- cyberpunk dark technical atmosphere
- premium product clarity
- proof/artifact visual language
- restrained neon accents
- mascot/signature identity

The site should feel:

```text
serious
sharp
technical
dark
minimal
opinionated
public-build
not corporate
not childish
not noisy
```

### References to avoid

Avoid directly imitating:

- CryptoPunks NFT collection style
- generic neon cyberpunk overload
- gaming mascot UI
- corporate AI SaaS gradients
- cartoon animal branding

### References to keep conceptually

Use these as abstract direction only:

```text
cypherpunk:
  privacy
  protocols
  keys
  proof
  resistance
  local-first trust

cyberpunk:
  dark city / tech-noir mood
  terminals
  networks
  neon restraint
  high-tech, low-bloat attitude

punk:
  independent
  anti-bloat
  sharp
  DIY
  non-corporate
```

## 4. Mascot direction

### Name

```text
Punk Porcupine
```

### Role

The mascot is the visible identity of `punk`.

It should work as:

- landing hero visual
- guide-agent avatar
- favicon / app icon
- social avatar
- future module badge system
- public-build storytelling character

### Symbolic meaning

```text
quills = boundaries / scope
dark body = local-first kernel
bright eye = inspection / intelligence
magenta quills = punk energy
cypherpunk details = proofs / keys / protocols
```

### Current direction

Use the strongest existing direction as reference:

```text
dark side-view porcupine
strong silhouette
neon magenta quill crown
single bright eye
serious disciplined expression
minimal cypherpunk/cyberpunk detail
```

### Final asset requirements

Create three mascot assets:

```text
A. Full mascot
   Use: hero, storytelling, public-build graphics

B. Head-only mark
   Use: logo, guide avatar, social avatar

C. Tiny signet
   Use: favicon, CLI/docs icon, 16–32px UI
```

### Manual vector reduction rules

For the final brand asset:

```text
- keep the strong side-view silhouette
- simplify the body to large geometric shapes
- simplify quills to 6–9 readable spikes
- preserve one bright eye
- keep magenta quills as signature
- allow at most 1–2 subtle protocol/key details
- remove busy circuit patterns
- remove armor-like details
- reduce anger; keep disciplined intelligence
- test at 128px, 64px, 32px, and 16px
```

### Illustrator brief

```text
Create a simplified vector mascot based on the selected cyberpunk/cypherpunk porcupine direction.

Dark graphite side-view porcupine, compact body, sharp neon magenta quill crown, one bright off-white intelligent eye, serious but calm disciplined expression.

Use clean geometric vector shapes and a strong readable silhouette. Add at most one or two subtle protocol-line or encryption-key motifs in the quills. Keep details restrained.

Premium dark tech brand. Not cute. Not goofy. Not angry. Not robotic. Not cyber armor. Not NFT collectible style. Not a human punk.
```

### Head-only mark brief

```text
Create a simplified head-only logo mark from the selected porcupine mascot direction.

Use a dark graphite angular porcupine head in side view, a sharp neon magenta quill crown, and one bright off-white intelligent eye. Keep the silhouette strong and readable at small sizes.

Use minimal geometric vector shapes and a premium cyberpunk/cypherpunk mood. Include at most one subtle protocol-line detail in the quills. No body, no armor, no busy circuits, not cute, not goofy, not angry, not NFT collectible style.
```

## 5. Visual system

### Palette

Suggested palette:

```text
Background:
  #07080B    near-black
  #0D1117    graphite black

Surfaces:
  #111820    dark slate
  #171E27    card surface
  #222B36    border / inactive detail

Text:
  #F4F1EA    warm off-white
  #B9C0C8    muted text
  #6E7884    secondary metadata

Accent:
  #FF2BD6    neon magenta
  #B615A8    deep magenta
  #7DF9FF    restrained cyan, optional only
  #F4C542    amber warning/accent, rare
```

Rules:

```text
- magenta is the signature accent
- cyan should be rare
- avoid rainbow neon
- dark surfaces should remain readable
- texture should be subtle, not noisy
```

### Typography

Suggested roles:

```text
Headline:
  sharp modern grotesk / technical sans

Body:
  readable modern sans

Artifacts / code:
  monospace
```

Tone:

```text
short claims
clear explanations
artifact captions
terminal/proof snippets
no purple prose
```

### Shapes

Use:

```text
rectangular cards
thin borders
subtle pixel cuts
small terminal panels
contract/proof receipts
dark hero capsule
```

Avoid:

```text
large glassmorphism blur
AI SaaS gradient blobs
overly rounded playful cards
busy cyberpunk textures
```

## 7.7 Punk Guide section

### Heading

```text
Ask the Punk Guide.
```

### Purpose

A future constrained guide agent explains the project and routes interested visitors.

### Scope

The guide can answer:

```text
What is punk?
Why core-first?
What are Punk Laws?
What is project memory?
What are DevPunk and PubPunk?
How can I follow the build?
How can I reach out?
```

The guide cannot:

```text
solve arbitrary tasks
write code for visitors
give general AI advice
access private project data
pretend the product is complete
execute tools
```

### UI concept

```text
┌─────────────────────────────┐
│ PUNK GUIDE                  │
│                             │
│ Ask about:                  │
│ • What is punk?             │
│ • Why core-first?           │
```

For the first landing, this can be a visual mock only.

Do not build the live agent before the site narrative is clear.

## 8. Guide agent guardrails

When implemented, the guide should be narrow.

### Allowed topics

```text
punk concept
roadmap
public build
core laws
modules
early collaboration
```

### Forbidden topics

```text
arbitrary user tasks
general coding help
private/internal details
legal/investment advice
medical/financial advice
tool execution
unverified roadmap promises
```

### Behavior

```text
short answers
honest status
no overclaiming
route to roadmap/public build/contact
```

### Example answer

User:

```text
Can punk build my app today?
```

Guide:

```text
Not yet. The first public build focuses on the core: flow, evals, contracts, gate, proof, and project memory. The point is to make agent work reliable before expanding capabilities.
```

## 12. Landing v0 acceptance criteria

The landing design is acceptable when:

```text
- the concept is clear in 5 seconds
- it does not overclaim product readiness
- the mascot reads at small size
- the visual language feels cypherpunk/cyberpunk but premium
- CTAs match current reality
- PubPunk is framed as future module
- the guide agent is scoped, not general-purpose
- public build story connects to repo artifacts
```
