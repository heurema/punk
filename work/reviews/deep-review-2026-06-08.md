# Punk Deep-Review — Unified Final Report

**Project:** punk — Rust trust-kernel for bounded AI work
**Core invariant:** contract → evidence → gate → proof; **only the gate decides**
**Reviewers:** Architect, Security, Correctness, Quality (lead)
**Branch reviewed:** `chore/repo-governance-v0.4.0`

---

## Executive Summary

punk is a deliberately side-effect-bounded, early-phase Rust workspace (13 crates) whose central architectural promise — that a single authority (the gate) writes decisions, and that acceptance requires both an accepting gate decision *and* a matching proof — **holds in code, structurally, not merely by convention.** Final-decision authority is asserted in exactly one place (`punk-gate`'s `gate_decision_boundary()` setting `writes_final_decision: true`); every other crate's truth-bearing boundary flag is a hardcoded-`false` const fn with no mutator. Layering is clean and acyclic, the CLI is a genuinely thin adapter, and the few active write slices (proofpack/module-host/events/project-manifest writers) are carefully hardened (create_new/no-clobber, symlink rejection, `.punk`-scoping, exact-byte readback).

Build and test health is excellent: `cargo check --workspace` and the full test run are green (392 tests pass, 0 fail), with **zero warnings** across a 13-crate workspace.

The review is nonetheless **mixed-leaning-solid** because the project ships method *names* and *docs* that imply integrity guarantees the code does not yet make. All four reviewers independently converged on the same fault line: **the two places that actually adjudicate truth — the proof-acceptance path and the governance enforcement layer — carry every finding capable of producing a wrong or forged decision.** Acceptance readiness checks structural ref *presence*, not hash *content*; the sole filesystem trust boundary can be escaped via an intermediate-directory symlink; and three of four governance checkers are honor-system only (no CI, no working pre-commit hook).

**Overall risk verdict: ACCEPTABLE FOR CURRENT PHASE, with conditions.** Nothing is exploitable today because no runtime consumes these seams. But the load-bearing trust seams are currently carrying *structural shape* where the product promises *verified linkage*. These must be closed — and the governance gates actually wired — before any proof writer, acceptance verifier, or publishing-write path is activated. Until then the code is honest about its parked scope; the risk is that the naming invites downstream over-trust.

**Severity tally:** 8 high · 19 medium · 59 low · 23 info (109 findings total)

## Prioritized Cross-Cutting Issues (deduplicated, highest severity first)

The most striking signal of this review is **convergence**: the top findings were each raised independently by 2–4 roles. They are deduplicated here into themes.

### CC-1 [HIGH] — Proof acceptance verifies structural ref presence, not hash content
*Raised by all four roles.*
`is_matching_proof_ready_for_acceptance` reports a proof "ready" when every required `(kind, ref)` carries *some* syntactically valid sha256 digest — it never recomputes or compares the hash against artifact content (`satisfies_requirement` compares kind+ref only). A proofpack with deliberately wrong/forged hashes reads as ready and feeds `positive_acceptance_preconditions_met`. The content-verifying `ProofpackWriterHashReferenceIntegrationModel` exists but is **off the public acceptance path**.
- `crates/punk-proof/src/lib.rs:263-265, 510-525, 531-536`
- Compounding: `verified()` constructor is circular (sets expected==observed from one input) `:8676-8684`; self-digest freshness check is tautological (recomputes over stored body, never compared to live proofpack) `:830, 8882-8889`; integration model treats empty evidence as Ready `:8847, 8896-8907`.
**Fix:** route acceptance through the content-verifying model and bind to a real `Accepted` `GateDecision`, OR rename to reflect structural-only readiness + add rustdoc and an explicit "self-asserted, not verified" marker. Add a test pinning the wrong-hash-still-ready limitation.

### CC-2 [HIGH] — Intermediate-directory symlink escapes repo root (path traversal / arbitrary file read)
*Raised by Security (reproduced live against the filesystem).*
`compute_artifact_file_digest` — the project's **sole filesystem trust boundary** — guards only the *final* path component against symlinks. A symlinked intermediate directory transparently escapes `repo_root` past the purely lexical `starts_with` check, forging the digest evidence acceptance will depend on.
- `crates/punk-core/src/lib.rs:253-272, 409-420`
**Fix:** canonicalize-and-reassert containment, or per-component `O_NOFOLLOW` walk.

### CC-3 [HIGH] — No single authority cross-validates GateDecision + Proofpack
*Raised by Architect, Security, Correctness.*
Proof binds to a gate decision by **opaque string equality** only; contract/receipt refs are never cross-validated against the gate, and acceptance reduces to two caller-supplied booleans. No component takes a real `GateDecision` + `Proofpack` and verifies accepting outcome, matching id, and ref set-equality. A typo'd ref yields a dangling requirement satisfiable by an equally typo'd digest; the linkage has no typed identity, dedup, or dangling-ref/well-formedness validation.
- `crates/punk-proof/src/lib.rs:469-536, 9399-9407`; `crates/punk-gate/src/lib.rs:192`
**Fix:** introduce a single cross-crate acceptance authority that verifies content hashes plus ref set-equality against the real decision.

### CC-4 [HIGH] — Governance checkers are honor-system: no CI, no working pre-commit
*Raised by all four roles.*
AGENTS.md advertises research-gate, work-ledger, and module-control as "Required checks" and says "always run," but **only `check_docs_governance.py` runs in CI**; the local pre-commit hook is a configless prek shim that no-ops. So a research-gate bypass, a parked-as-active flag flip, or a dangling memory-link can land unchecked — exactly the RESEARCH-GATE BYPASS and HIDDEN-RUNNER-AIDS risks the docs themselves warn about. Shared docs even overstate that `check.sh` routes them.
- `scripts/check.sh:14-60`; `.github/workflows/docs-governance.yml:42-46`; `.git/hooks/pre-commit:13`; `AGENTS.md:41-42`
**Fix:** stand up real CI enforcement for the three unrun checkers.

### CC-5 [HIGH] — DocImpact parser is brittle and misfires on legitimate, real reports
*Raised by Correctness (empirically confirmed against existing reports).*
The DocImpact validator requires **exactly 2-space indentation** and only one exact heading spelling, and its allowed-classification enum has **drifted from established reporting practice.** Confirmed live: it falsely fails 4 real reports as `DOC_IMPACT_BLOCK_MISSING` and 5 as `DOC_IMPACT_INVALID_CLASSIFICATION`. Because it gates real commits via auto-derived `report_paths`, this is the **most actively harmful defect today** — it blocks valid changes. Separately, DocImpact is a hard CI failure for any `crates/` change despite DOC-GOVERNANCE.md calling it advisory-only at Level 0.
- `scripts/check_docs_governance.py:343-355, 39-47, 49-57`; `docs/_schema/doc-impact.v0.1.schema.yaml:8-16`; `docs/product/DOC-GOVERNANCE.md:78-84,118`
**Fix:** tolerant heading/indent parsing; reconcile the enum with practice; make the Level-0 gate advisory per its own owner doc.

### CC-6 [HIGH] — Boundary eval cases assert against hardcoded-false constants (circular oracle)
*Raised by Quality, echoed by Architect/Correctness.*
~40 of the eval suite's core architectural-isolation assertions (`does_not_write_gate_decision` / `creates_proofpack` / `invokes_writer`) read **compile-time-constant `false` flags.** The suite therefore emits a green "core invariant holds" signal while observing no actual side effect — it can never fail and **cannot detect the decision/proof drift it claims to protect against.** The same harness's writer-slice cases do the right thing by probing the real filesystem (`temp_path.join(".punk/decisions").exists()`), proving the fix pattern.
- `crates/punk-eval/src/lib.rs:7897-7971`
**Fix:** treat the const-false flags as the thing under audit, not the oracle; add ≥1 genuine side-effect-observing case per invariant.

### CC-7 [HIGH] — `GateOrPolicyApproval` precondition self-asserted from ref presence
*Raised by Architect.*
The precondition literally named `GateOrPolicyApproval` is marked "covered" by structural ref-presence and is never bound to a real `Accepted` `GateDecision`. The name promises approval semantics the code does not enforce, risking a future runtime treating a `Ready` proposal as gate-approved before any `GateDecision` exists.
- `crates/punk-module-host/src/lib.rs:5410-5419`
**Fix:** bind to a real accepting decision, or rename + mark self-asserted-not-verified.

### CC-8 [MEDIUM] — Governance checks that *do* run are trivially evadable
*Security, with Correctness/Quality support.*
- Module-control markers use case-sensitive, whitespace-exact **substring** matching: `auto_apply: true` is bypassed by `auto_apply : true`, `True`, or `=true` — passing the check does not prove the no-auto-apply / no-writer / no-secrets boundaries hold. `scripts/check_module_control_markers.py:15-57`
- DocImpact accepts a **self-asserted classification** never reconciled against the changed file set: a diff touching `crates/` + `ARCHITECTURE.md` can self-label `classification:none` and pass. `scripts/check_docs_governance.py:845-883`
- Active-CLI-surface checker's fixed **8-line future-suppression window** lets unimplemented commands be described in present-tense run/today verbs. `:648-703`
- `check_research_gate.py` never enforces the R1+ "must carry research refs" rule — an R1/R2/R3 goal with `required:true` and empty refs passes. `:127-173`
- Work-ledger existence check covers only `report_refs`/`decision_refs`; dangling `contract_refs`/`proof_refs` pass silently. `scripts/check_work_ledger.py:154-168`
**Fix:** structural (not substring) marker parsing; reconcile self-asserted fields against the actual diff.

### CC-9 [MEDIUM] — Proof `artifact_digests` accepts duplicate/conflicting `(kind,ref)` entries
*Security + Architect.*
Two entries with the same `(kind, ref)` but different hashes both satisfy the requirement, letting a contradictory manifest read as complete. Manifest digests also depend on insertion order, weakening replay/tamper comparison.
- `crates/punk-proof/src/lib.rs:394-397, 510-525`

### CC-10 [MEDIUM] — Flow state machine cannot enforce proof-before-acceptance
*Architect, Correctness, Quality.*
`(DecisionWritten, MarkReported) -> Reported -> Closed` reaches terminal "done" with **no `ProofWritten`**; the machine carries no gate-outcome/decision-polarity concept, so the canonical proof gate cannot be expressed in the lifecycle graph. Compounding: the entire **back half of the lifecycle** (decision/proof/report/close, block/escalate/cancel) is **untested.**
- `crates/punk-flow/src/lib.rs:455-467, 507-561, 519-521`
*Correctly a deferred Phase-1 gap to revisit when gate/proof wiring lands — but the missing tests are not deferrable.*

### CC-11 [MEDIUM] — Eval harness flakiness: cleanup folded into pass; 175 cases → 6 tests
*Correctness, Architect, Quality.*
Temp-dir cleanup success is folded into each case's pass condition (`crates/punk-eval/src/lib.rs:5627, 5923, 6073, ...`), so an OS cleanup failure flips the nominally-deterministic smoke suite to FAIL. Separately, 175 logical assertions collapse into 6 cargo tests via a hand-maintained 175-case vector (`:475-656, 13794-13822, 13810`) that can silently drift from the `eval_*` function set, erasing per-case CI granularity.

### CC-12 [MEDIUM] — CRATE-STATUS.md labels empty `punk-rules` stub "active-core"
*Quality, Correctness, Security, governance cluster — raised by three roles.*
A 3-line, zero-test skeleton is presented as required-for-stability, contradicting the doc's own legend and a later admission it is a skeleton. `punk-rules` must **not** be modeled as an active trust surface (it has 0 tests yet ships an active test executable).

## Findings by Role

### Architect — verdict: **solid**

Across all seven clusters the load-bearing core invariants hold in code, structurally. Final-decision authority is asserted in exactly one place; every other crate's boundary flags (`writes_final_decision`, `creates_acceptance_claim`, `absorbs_evidence`, `invokes_writer`, `writes_gate_decision`) are hardcoded `false` in const fns with no mutators. The acceptance closure is correctly conjunctive: `positive_acceptance_preconditions_met` requires **both** an accepting gate decision and a matching proof, with neither sufficient alone. Layering is clean and acyclic — `punk-gate` has zero internal deps, proof depends only on `punk-core`, eval is a correct leaf consumer, the CLI never touches `.punk/decisions`, `.punk/proofs`, or `.punk/runs`. No path was found where a non-gate component writes a decision, a module decides, or an adapter owns truth.

The concerns cluster into one architectural theme — the system's central promise is *verifiable linkage*, yet several seams that should carry verification currently carry only structural shape (CC-1, CC-3, CC-7) — plus engineering-quality observations: governance enforcement credibility gap (CC-4), DocImpact overruling its own owner doc, three disagreeing precedence lists, the eval-as-data harness collapsing 175 assertions into 6 tests, duplicated safe-ref policy and 8×/12× hand-duplicated finding/validation/boundary-flag families in pubpunk/module-host, and the 14k-line proof module. **Fix posture:** close the proof-integrity and gate-approval naming/binding gaps first; then wire the governance gates; then consolidate duplicated safe-ref policy and finding boilerplate before a shared runtime consumes these types. The flow proof-before-acceptance gap is correctly deferred to Phase 1.

### Security — verdict: **mixed**

From a classic-attack-surface standpoint the codebase is clean: overwhelmingly pure, dependency-light (only third-party crate is `sha2 0.10.9`, lockfile-pinned), no untrusted deserialization, no network I/O, no secret handling; authority-bearing structs honestly hardcoded `false` and test-asserted. The concern is the specific seams where untrusted input will enter once the documented downstream (real writer, acceptance verifier, activated governance) is wired in. The verdict is mixed because the project is honest about parked scope yet ships names/docs implying integrity it does not yet provide.

The most serious cluster is the **integrity chain as one connected attack path** (CC-2 → CC-1 → CC-3): an actor who can shape the repo tree can make a digest attest to out-of-repo bytes (intermediate-symlink traversal, reproduced live), attach any syntactically valid hash to a proofpack, declare an arbitrary ref set, and have the public acceptance predicate treat it as ready. The **governance cluster** (CC-4, CC-8) is the second axis: enforcement is materially weaker than the canonical docs claim, and the checks that exist are evadable. **Recommendation:** before any writer/verifier/publishing path activates — (1) fix the intermediate-symlink traversal, (2) introduce a single cross-crate acceptance authority verifying content hashes + ref set-equality, (3) stand up real CI enforcement with structural marker parsing. Everything else is defense-in-depth. `punk-rules` is confirmed an empty skeleton and must not be modeled as an active trust surface.

### Correctness — verdict: **mixed**

Correctness splits cleanly: the decision-bearing Rust kernel is overwhelmingly sound, while the two places that adjudicate truth — proof acceptance and governance scripts — carry the only defects capable of producing a wrong decision. Foundation, gate, modules, orchestration, and CLI emit no incorrect gate decision or unsafe write: deterministic, aggressive construction-time validation, fail-closed by default (`Ready` only when findings empty; writer results `Ready` only on terminal success), well-hardened write slices (absolute-root confinement, symlink rejection, `create_new`/`O_EXCL`, exact-byte readback). The `ApprovedForRun`-is-not-execution-permission invariant holds; documented unreachable defensive branches are correct layered defense.

The substantive risk lives in `punk-proof`'s acceptance closure (CC-1, plus circular `verified()` and tautological self-digest). The governance scripts are the most actively harmful *today* (CC-5: empirically confirmed false BLOCK_MISSING ×4 and false INVALID_CLASSIFICATION ×5; superseded-ref scanner matches inside fenced code blocks; inconsistent YAML-list parsing; dangling-ref-tolerant work-ledger). The eval suite is clean as production logic but its harness robustness is environment-dependent (CC-11) and its substring oracles can fail open. Orchestration observations are structural (CC-10; event-log torn-line sequence reuse). **Net:** the kernel earns trust; the proof-acceptance content-integrity gap and governance parser fragility must close before "proofs validate acceptance" and "docs-code fidelity" can be taken at face value.

### Quality (lead reviewer) — verdict: **solid**

punk is a healthy, deliberately-scoped early-phase workspace; the dominant signal is consistency between canonical docs and code. Boundary methods are uniformly hardcoded `false`, parked scope is generally marked not-active, the small kernels are clean and faithful to PUNK-LAWS, the narrow active write slices are carefully guarded and accurately documented. Build is clean with meaningful test mass. Three themes to track before scaling:

1. **Trustworthiness of the verification layer itself** — the circular boundary-eval oracle (CC-6) paired with the acceptance-name doc-fidelity gap (CC-1). The fix is to make the const-false flags the thing under audit and add real side-effect-observing cases, plus rustdoc + a wrong-hash-still-ready test.
2. **Enforcement-vs-aspiration in governance** (CC-4, CC-8) — PUNK-LAWS research/dogfooding discipline rests on manual compliance; the good docs-governance checker can be silenced by a stray `parked`/`not` token and does not cover crate-status rows (hence the `punk-rules` mislabel, CC-12).
3. **Maintainability debt** — 10–14.5k-line single-file crates with copy-paste; 3–4× duplicated safe-ref predicate; the two-place 1400-char string; the hand-maintained magic 175 count; and real coverage gaps on correctness-critical paths (entire back half of flow lifecycle, `punk-gate` non-happy paths, no end-to-end acceptance-closure integration suite).

**Net:** the no-write/no-decide boundary holds in code today; the work to schedule is making the tests and governance *prove* it rather than *assert* it, hardening research-gate enforcement, and decomposing the monolithic crates before more handoff packets land.

# Appendix: All confirmed findings (machine list)

## [HIGH] foundation/security — Intermediate-directory symlink escapes repo root in file-digest verification (path traversal / arbitrary file read)
- **location:** crates/punk-core/src/lib.rs:253-272 (compute_artifact_file_digest) and :409-420 (artifact_path)
- **verdict:** confirmed
- **description:** artifact_path() joins repo_root with the validated ref and checks `path.starts_with(repo_root)` purely lexically; validate_repo_relative_artifact_ref() blocks `..`, leading `/`, `~`, and backslashes, so the lexical check always passes. compute_artifact_file_digest() then calls fs::symlink_metadata() on the FINAL path only and rejects the read if that final component is a symlink. It does NOT detect a symlink on an INTERMEDIATE component. If any directory in the ref (e.g. `artifacts/`) is itself a symlink pointing outside the repo (or to an absolute path), the OS transparently resolves it: symlink_metadata on `artifacts/passwd` reports a regular file, and fs::read returns bytes from outside repo_root. I reproduced this against the live filesystem: with repo/artifacts -> /tmp/secret, ref `artifacts/passwd` passed starts_with, reported final-component is_symlink=false / is_file=true, and read the out-of-repo file content. The SymlinkNotSupported guard and the symlink unit tests (lines 843-891) only cover the final-component case, giving false assurance.
