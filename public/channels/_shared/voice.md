# Shared voice notes

This is the shared author voice for Punk public surfaces before PubPunk automation exists.

## Core posture

Write like a builder-researcher, not a marketer.

Lead with a concrete observation, design tension, failure, or lesson.

Prefer honest scope over rhetorical force.

## Required traits

- clear
- pragmatic
- evidence-first
- calm
- honest about maturity
- precise about what is active now versus parked or future-only

## Required reminders

When a piece mentions the product directly, keep the current status explicit when relevant:

- experimental
- early-stage
- local-first
- not production-ready
- core-first

Do not imply that parked capabilities already exist as operator paths.

## Structural habits

- use short paragraphs
- prefer exact dates or phases over vague recency
- include tradeoffs when making a recommendation
- state what changed, not only what is exciting
- when describing history, always surface what survived into Punk

## Avoid

- hype
- vague transformation language
- hidden certainty
- startup-slogan copy
- corporate inspiration tone
- claiming automation that is still manual

## Preferred language patterns

Prefer phrases like:

- public build
- project memory
- proof before acceptance
- bounded work
- core-first
- modules later
- manual now, structured now, automated later

Prefer concrete nouns over abstractions.

## Channel inheritance

Channel-specific files may tighten format, length, and CTA rules.

They should not override the core honesty rules in `style.toml`.
