# Manual now, structured now, automated later

## Core idea

The publication structure should exist before the publication module exists.

That way the future module adopts real project memory instead of inventing a second content store.

## Why it matters

Public narrative is part of project memory.

If stories, posts, receipts, and metrics live only in chat or inside a future tool, the public build loses continuity.

## Narrative beats

1. The project is public from day zero.
2. The Journal is a structured chronology, not generic marketing content.
3. Stories, posts, channel rules, receipts, and metrics are tracked manually first.
4. Future automation should read this structure, not replace it.

## Current status

Manual publication is allowed now.

Publication receipts and metrics snapshots are repo-tracked.

PubPunk automation inside `punk` is still parked.

## What stays out of scope

- automated posting
- metrics scraping
- scheduling
- growth automation
