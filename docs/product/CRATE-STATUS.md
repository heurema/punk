# Crate Status

## Status vocabulary

- `active-core` — required for the current stability target
- `incubating` — tested but not default/user-facing
- `parked` — boundary exists; minimal stub/docs only
- `retired` — removed or legacy-only

## Current CLI surface

Currently implemented CLI commands:

- `punk init <project-id>`
- `punk init <project-id> --mode brownfield`
- `punk flow inspect`
- `punk publishing locate [--project-root <path>] [--json]`
- `punk eval run smoke`
- `punk eval run smoke --format json`

`punk init <project-id>` is current CLI behavior for default greenfield Dogfooding Level 0 compact manual project-memory scaffolding.
It initializes the current directory in place and does not create a new subdirectory named `<project-id>`.
It records `project_id`, `entry_mode = greenfield`, `[memory] layout = "compact"`, inactive `[runtime]` metadata, and inactive `[instructions]` view metadata, then creates `.punk/README.md`, `.punk/project.toml`, `.punk/memory/` as tracked durable memory, and `.punk/instructions/` as thin source instruction pages. It does not activate root-level dogfooding layout, brownfield reconstruction, grayfield reconciliation, generated `.punk/views`, `.punk/` runtime storage, flow/event persistence, contract writing, receipt writing, gate writing, proof writing, or proofpack writing.

`punk init <project-id> --mode brownfield` is current CLI behavior only for a brownfield entry scaffold.
It records `entry_mode = brownfield`, `reconstruction_status = not_started`, and advisory-only authority, then creates empty `.punk/memory/reconstruction/` placeholders plus the same thin `.punk/instructions/` source pages.
It does not scan the repository, reconstruct knowledge, generate summaries, generate contracts/specs, accept claims, write runtime state, or prove anything.

`punk publishing locate [--project-root <path>] [--json]` is current CLI behavior only for resolving a local publishing workspace binding.
It reads `.punk/publishing.toml` and local-only `.punk/publishing.local.toml`, validates that the logical workspace refs match, expands an absolute or home-relative local workspace pointer, and reports located/blocked status.
It does not write files, publish, open browsers, call APIs, read credentials, activate adapters, make external platforms source of truth, create receipts, or run PubPunk automation.
This locator is a transitional exception, not a precedent for putting PubPunk behavior into `punk-project`, `punk-cli`, or active core. Publishing inventory, drafting, planning, receipt creation, and external publication remain module work behind future PubPunk/module-host promotion.

Punk may run validators in the user's environment, but the user's executor, provider, prompt, skill, or local model setup must not become active-core authority.

Executor-specific behavior and semantic assessor command interfaces remain parked until the roadmap explicitly promotes DevPunk, adapters, or an assessor boundary.

## Parked module host note

`punk-module-host` is not a plugin runtime while incubating.

The incubating crate may model pure invocation envelopes and assessment wrapping. It must not add a Wasm runtime dependency, plugin loader, plugin installer, plugin marketplace, dynamic module execution, adapter invocation, or public `punk plugin` command until a later ADR promotes that scope.
